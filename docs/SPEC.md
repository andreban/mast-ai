# Technical Specification — MAST

## Architecture

```
┌────────────────────────────────────────────────────┐
│                     Browser                        │
│                                                    │
│  ┌──────────────┐  runs   ┌──────────────────────┐ │
│  │  AgentRunner │────────▶│   AgentConfig        │ │
│  │              │         └──────────────────────┘ │
│  │  holds       │         (pure data blueprint)     │
│  └──────┬───────┘                                  │
│         │ holds                                    │
│  ┌──────▼───────┐                                  │
│  │  LlmAdapter  │  (interface)                     │
│  └──────┬───────┘                                  │
│         │ implements                               │
│  ┌──────┴──────────────────────┐                   │
│  ▼                            ▼                   │
│  UrpAdapter          PromptApiAdapter             │
│  (Hybrid Mode)       (Client-Side Mode)           │
│       │              Chrome Prompt API            │
└───────┼───────────────────────────────────────────┘
        │ HTTP POST / WebSocket
┌───────▼────────────────────────────────────────────┐
│  Remote Backend (any language — Go, Rust, Python)  │
│  Implements the Universal Remote Protocol (URP)    │
└────────────────────────────────────────────────────┘
```

The library is a single TypeScript package (`MAST`). All provider adapters live in `src/adapter/`. Agent logic lives in `src/agent.ts` and `src/runner.ts`. The `LlmAdapter` interface in `src/adapter/index.ts` is the extension point.

`AgentConfig` is a pure data blueprint (name, instructions, optional output schema, tool names) with no model reference. `AgentRunner` owns the `LlmAdapter` and is the execution engine. The same runner can execute multiple agents; the same agent config can be run by different runners backed by different adapters.

The agentic loop always runs in the browser. The browser is the source of truth for conversation state. Remote backends are stateless reasoning engines — they receive the full conversation on every request and return either a text response or tool call instructions.

---

## Execution Modes

### Client-Side Mode
The `LlmAdapter` is backed by a browser-native model — the primary example being the [Chrome Prompt API](https://developer.chrome.com/docs/ai/prompt-api) (`PromptApiAdapter`), which runs Gemini Nano on-device via `window.LanguageModel`. No network requests are made for inference. Tools execute in the browser. The `AgentRunner` API is identical regardless of mode.

### Hybrid Mode (Primary Focus)
The `LlmAdapter` is a `UrpAdapter`, which serialises conversation state and forwards it to a remote endpoint over HTTP or WebSockets. The browser still owns the loop: after each remote call the runner checks for tool calls, executes them locally, and injects the results before the next remote call.

### Server-Side Mode
Out of scope for this library. The agent loop runs on the server; the browser communicates with it via a bespoke API. `MAST` does not provide a server-side runner.

## Core Types

### `Message` / `Role` / `MessageContent` (`src/types.ts`)

```typescript
export type Role = 'user' | 'assistant';

export type MessageContent =
  | { type: 'text'; text: string }
  | { type: 'tool_calls'; calls: ToolCall[] }
  | { type: 'tool_result'; id: string; name: string; result: unknown };

export interface Message {
  role: Role;
  content: MessageContent;
}
```

`MessageContent` is a discriminated union. A single assistant turn that requests one or more tools uses `type: 'tool_calls'`. Each tool result is a separate user-role message with `type: 'tool_result'`. The runner inserts these automatically; callers never construct tool messages manually.

### `ToolCall` (`src/types.ts`)

```typescript
export interface ToolCall {
  id: string;       // opaque identifier echoed back to the adapter
  name: string;
  args: unknown;    // parsed JSON object matching the tool's parameters schema
}
```

### `AgentEvent` (`src/types.ts`)

```typescript
export type AgentEvent =
  | { type: 'tool_call_started';   name: string; args: unknown }
  | { type: 'tool_call_completed'; name: string; result: unknown }
  | { type: 'text_delta';          delta: string }
  | { type: 'thinking';            delta: string }
  | { type: 'done';                output: string; history: Message[] };
```

The public event type yielded by `AgentRunner.runStream`. `tool_call_started` / `tool_call_completed` events are generated by the runner. `text_delta` and `thinking` are forwarded from the adapter when streaming is supported. Models supporting extended thinking (e.g. reasoning tokens) emit `thinking` events. `done` is emitted once with the fully assembled output text and the complete updated `history` (including the new user turn, any tool messages, and the final assistant turn) after the loop exits.

### `AgentConfig` (`src/agent.ts`)

```typescript
export interface AgentConfig {
  name: string;
  instructions: string;
  tools?: string[];                 // names of tools the agent may invoke
  outputSchema?: Record<string, unknown>;  // JSON Schema for structured output
}
```

A plain serialisable object — no class, no methods. `tools` is a list of names; full `ToolDefinition` objects are owned by the `ToolRegistry` and resolved at run time. `AgentConfig` values can be stored in JSON/YAML files and loaded at runtime. An optional `createAgent(config: AgentConfig): AgentConfig` helper validates the config at construction time.

`outputSchema`, when set, is forwarded to the adapter via `AdapterRequest.outputSchema`. Adapters that do not support structured output ignore the field silently.

### `ToolDefinition` / `Tool` / `ToolRegistry` (`src/tool.ts`)

```typescript
export interface ToolDefinition {
  name: string;
  description: string;
  parameters: Record<string, unknown>;  // JSON Schema object
}

export interface Tool<TArgs = unknown, TResult = unknown> {
  definition(): ToolDefinition;
  call(args: TArgs, context: ToolContext): Promise<TResult>;
}

export interface ToolContext {
  signal?: AbortSignal;  // forwarded from the runner; allows long-running tools to be cancelled
}

export class ToolRegistry {
  register(tool: Tool): this;           // builder-style
  get(name: string): Tool | undefined;
  definitions(): ToolDefinition[];
}
```

`ToolDefinition` is the runtime metadata forwarded to the remote backend in a URP request. It is never stored inside `AgentConfig`.


### `LlmAdapter` (`src/adapter/index.ts`)

```typescript
export interface AdapterRequest {
  messages: Message[];
  system?: string;
  tools: ToolDefinition[];
  outputSchema?: Record<string, unknown>;  // JSON Schema
  config?: ModelConfig;
}

export interface AdapterResponse {
  text?: string;            // undefined when tool calls were issued
  toolCalls: ToolCall[];    // empty on a final text response
}
```typescript
export interface AdapterStreamChunk {
  type: 'text_delta' | 'tool_call' | 'thinking';
  delta?: string;           // present when type === 'text_delta' or 'thinking'
  toolCall?: ToolCall;      // present when type === 'tool_call'
}

export interface LlmAdapter {
  generate(request: AdapterRequest): Promise<AdapterResponse>;
  generateStream?(request: AdapterRequest): AsyncIterable<AdapterStreamChunk>;
}
```

Adapters that do not implement `generateStream` receive a fallback: the runner calls `generate` and emits the result as a single `text_delta` chunk followed by `done`. Adapters that support true streaming override `generateStream`.

`text` and `toolCalls` on `AdapterResponse` are mutually exclusive per turn, matching the LLM turn model.

### `ModelConfig` (`src/adapter/index.ts`)

```typescript
export interface ModelConfig {
  temperature?: number;
  maxTokens?: number;
  topP?: number;
  stopSequences?: string[];
  [key: string]: unknown;   // pass-through for provider-specific fields
}
```

Passed through to the adapter unchanged. Adapters map fields to provider-specific request parameters.

### `AgentRunner` (`src/runner.ts`)

```typescript
export interface AgentResult {
  output: string;
}

export class AgentRunner {
  constructor(adapter: LlmAdapter, registry?: ToolRegistry);

  /** Creates a Conversation that automatically tracks history across turns. */
  conversation(agent: AgentConfig): Conversation;

  /** Primary entry point for manual multi-turn use. */
  runBuilder(agent: AgentConfig): RunBuilder;

  /** Single-turn convenience methods (delegate to runBuilder). */
  runStream(agent: AgentConfig, input: string): AsyncIterable<AgentEvent>;
  run(agent: AgentConfig, input: string): Promise<AgentResult>;
  runTyped<T>(agent: AgentConfig, input: string): Promise<T>;
}
```

Owns the `LlmAdapter` and an optional `ToolRegistry`. Stateless: all conversation history is carried by the caller (or by a `Conversation` wrapper). The same runner can execute multiple agents; the same `AgentConfig` can be passed to runners with different adapters.

The agentic loop lives in `RunBuilder.runStream`:

1. Validate that all names in `agent.tools` exist in the registry. Throw `AgentError` if any are missing.
2. Prepend `history` to the new user `Message` and construct an `AdapterRequest` with resolved `ToolDefinition`s.
3. Call `adapter.generateStream` (or fall back to `generate`) and forward `text_delta` chunks as `AgentEvent`.
4. Collect `tool_call` chunks. When the turn ends with tool calls, emit all `tool_call_started` events.
5. Execute all tool calls **concurrently** using `Promise.all`. Each tool receives a `ToolContext` containing the runner's `AbortSignal`.
6. Emit all `tool_call_completed` events in original call order.
7. Append assistant tool-call message and one user tool-result message per result to the internal history.
8. Repeat from step 3 until the model returns a text-only turn.
9. Emit `{ type: 'done', output }`.

### `RunBuilder` (`src/runner.ts`)

```typescript
export class RunBuilder {
  /** Prepend prior conversation turns. */
  history(messages: Message[]): this;

  /** Attach an AbortSignal to cancel the run and any in-flight tool calls. */
  signal(signal: AbortSignal): this;

  runStream(input: string): AsyncIterable<AgentEvent>;
  run(input: string): Promise<AgentResult>;
  runTyped<T>(input: string): Promise<T>;
}
```

Produced by `AgentRunner.runBuilder`. The builder is consumed by the execution method; callers obtain a fresh builder for each turn.

`run` is a convenience collector over `runStream` that concatenates all `text_delta` events into `AgentResult.output`.

`runTyped<T>` is a thin typed wrapper over `run` that parses the final output via `JSON.parse` and returns it as `T`. A parse failure throws `AgentError`.

**Multi-turn usage pattern (manual):**

```typescript
// First turn (no history).
const first = await runner.run(agent, 'My name is Alice.');

// Build history; caller is responsible for maintaining it across turns.
const history: Message[] = [
  { role: 'user',      content: { type: 'text', text: 'My name is Alice.' } },
  { role: 'assistant', content: { type: 'text', text: first.output } },
];

// Second turn with history.
const second = await runner
  .runBuilder(agent)
  .history(history)
  .run('What is my name?');
```

The runner remains stateless; the caller owns and extends the history. For the common case, prefer `Conversation` (see below) which handles history bookkeeping automatically.

### `Conversation` (`src/conversation.ts`)

```typescript
export class Conversation {
  /** Full conversation history, updated automatically after each completed turn.
   *  Mutate directly to trim or compress history before the next turn. */
  history: Message[];

  run(input: string, signal?: AbortSignal): Promise<AgentResult>;
  runStream(input: string, signal?: AbortSignal): AsyncIterable<AgentEvent>;
}
```

Created via `AgentRunner.conversation(agent)` — not constructed directly by callers.

`Conversation` wraps a `RunBuilder` pair and automatically appends the updated history after each completed turn. History is taken from the `done` event's `history` field, which includes the full turn: user message, any tool call/result pairs, and the final assistant message.

**`run`** delegates to `RunBuilder.runStream` internally, captures the `done` event's `history`, updates `this.history`, and returns `AgentResult`. If the run throws, `this.history` is left unchanged — the failed turn leaves no trace.

**`runStream`** returns an `AsyncIterable<AgentEvent>` that forwards all events from the underlying stream. History is updated only when the stream is consumed to completion (i.e. the `done` event is received). If the iterable is abandoned early, history is not modified — appending a partial reply would corrupt the conversation context.

**Trimming history:** Because `history` is a public mutable array, callers can trim or compress it between turns to stay within a model's context window:

```typescript
// Drop everything except the last 10 messages.
conv.history = conv.history.slice(-10);
```

**Multi-turn usage pattern (automatic):**

```typescript
const conv = runner.conversation(agent);

// Each call to run / runStream automatically maintains history.
await conv.run('My name is Alice.');
const result = await conv.run('What is my name?');
console.log(result.output); // "Your name is Alice."

// Streaming variant:
const conv2 = runner.conversation(agent);
for await (const event of conv2.runStream('Hello!', abortController.signal)) {
  if (event.type === 'text_delta') process.stdout.write(event.delta);
}
```

### `AgentExecutor` / `AgentTool` (`src/agent-tool.ts`)

Because a sub-agent is exposed to its parent as a plain `Tool`, the sub-agent's execution mode is entirely independent of the parent's. The unifying abstraction is `AgentExecutor` — anything that can run an agent given a string input:

```typescript
export interface AgentExecutor {
  run(input: string, context: ToolContext): Promise<{ output: string }>;
}
```

`AgentTool` wraps any `AgentExecutor` as a `Tool`, making the sub-agent's location invisible to the parent:

```typescript
export class AgentTool implements Tool {
  constructor(
    definition: ToolDefinition,  // name, description, parameters exposed to the parent adapter
    executor: AgentExecutor,
  );

  definition(): ToolDefinition;
  call(args: unknown, context: ToolContext): Promise<{ output: string }>;
}
```

**How `call` works:**

1. Serialises `args` to a JSON string and passes it to `executor.run(input, context)`.
2. `ToolContext` (including `signal`) is forwarded to the executor so cancellation propagates.
3. Returns `{ output }` so the parent adapter receives a structured, readable result.

The three sub-agent execution modes each have a corresponding `AgentExecutor` implementation:

| Sub-agent mode | `AgentExecutor` implementation |
|---|---|
| **Client-side** | `localAgent(config, runner)` — binds `AgentConfig` + `AgentRunner` |
| **Hybrid** | `localAgent(config, runner)` with `runner` using `UrpAdapter` |
| **Server-side** | `AcpAgent` — calls a remote ACP-compliant endpoint |

#### `localAgent` helper

Binds an `AgentConfig` and `AgentRunner` into an `AgentExecutor`. The sub-agent loop runs in the browser; the `LlmAdapter` on the runner determines whether inference is local (`PromptApiAdapter`) or remote (`UrpAdapter`).

```typescript
export function localAgent(agent: AgentConfig, runner: AgentRunner): AgentExecutor;
```

**Usage pattern (hybrid parent → client-side child):**

```typescript
const childRunner = new AgentRunner(new PromptApiAdapter());
const childAgent: AgentConfig = {
  name: 'Classifier',
  instructions: 'Classify the intent in the "text" field of the JSON input.',
};

const classifyTool = new AgentTool(
  {
    name: 'classify_intent',
    description: 'Classifies the user intent in a piece of text.',
    parameters: {
      type: 'object',
      properties: { text: { type: 'string' } },
      required: ['text'],
    },
  },
  localAgent(childAgent, childRunner),
);

const registry = new ToolRegistry().register(classifyTool);
const parentRunner = new AgentRunner(parentAdapter, registry);
```

#### `AcpAgent` — server-side sub-agents

`AcpAgent` implements `AgentExecutor` against the Agent Call Protocol (ACP), a minimal HTTP convention for server-side agent endpoints (see [Agent Call Protocol](#agent-call-protocol-acp) below).

```typescript
export interface AcpAgentOptions {
  url: string;
  headers?: Record<string, string>;
}

export class AcpAgent implements AgentExecutor {
  constructor(options: AcpAgentOptions);
  run(input: string, context: ToolContext): Promise<{ output: string }>;
}
```

**Usage pattern (hybrid parent → server-side child):**

```typescript
const summariseTool = new AgentTool(
  {
    name: 'summarise',
    description: 'Summarises a long piece of text using a server-side agent.',
    parameters: {
      type: 'object',
      properties: { text: { type: 'string' } },
      required: ['text'],
    },
  },
  new AcpAgent({ url: '/api/agents/summariser' }),
);
```

Custom `AgentExecutor` implementations can wrap third-party agent APIs (e.g. OpenAI Assistants, LangChain agents) without modifying `AgentTool`.

### `AgentError` (`src/error.ts`)

```typescript
export class AgentError extends Error {
  constructor(
    message: string,
    public readonly cause?: unknown,

`AdapterError` is thrown by adapter implementations for upstream failures (HTTP errors, malformed responses). `AgentError` is thrown by the runner for agent-level failures (missing tool, invalid output schema, cancelled run).

## Universal Remote Protocol (URP)

The URP is a language-agnostic JSON protocol that decouples the browser agent loop from the inference backend. Any server that speaks URP can act as the reasoning engine.

### Request Schema

`POST /` (or any path agreed by the developer)

```jsonc
{
  "messages": [
    {
      "role": "user",
      "content": { "type": "text", "text": "What is 2 + 2?" }
    }
  ],
  "available_tools": [
    {
      "name": "calculator",
      "description": "Evaluates a mathematical expression.",
      "parameters": {
        "type": "object",
        "properties": {
          "expression": { "type": "string" }
        },
        "required": ["expression"]
      }
    }
  ],
  "configuration": {
    "temperature": 0.7,
    "max_tokens": 1024
  }
}
```

| Field              | Type                   | Required | Notes                                                  |
|--------------------|------------------------|----------|--------------------------------------------------------|
| `messages`         | `Message[]`            | Yes      | Full conversation history, newest last                 |
| `available_tools`  | `ToolDefinition[]`     | Yes      | Metadata only — no handler code                        |
| `configuration`    | `object`               | No       | Provider-specific parameters; backends may ignore keys |

### Response Schema

```jsonc
{
  "text_content": "The answer is 4.",
  "tool_calls": [],
  "usage_metrics": {
    "input_tokens": 42,
    "output_tokens": 10
  }
}
```

| Field            | Type           | Required | Notes                                                      |
|------------------|----------------|----------|------------------------------------------------------------|
| `text_content`   | `string`       | No       | Present on final text turns; omitted when tool calls are issued |
| `tool_calls`     | `UrpToolCall[]`| Yes      | Empty array on text turns                                  |
| `usage_metrics`  | `object`       | No       | Optional token accounting                                  |

`text_content` and a non-empty `tool_calls` are mutually exclusive.

### `UrpToolCall`

```jsonc
{
  "id": "call_abc123",
  "name": "calculator",
  "arguments": { "expression": "2 + 2" }
}
```

### `UrpAdapter` (`src/adapter/urp.ts`)

```typescript
export interface UrpTransport {
  send(request: UrpRequest): Promise<UrpResponse>;
}

export class UrpAdapter implements LlmAdapter {
  constructor(transport: UrpTransport);
  generate(request: AdapterRequest): Promise<AdapterResponse>;
}
```

`UrpAdapter` translates `AdapterRequest` → `UrpRequest`, calls `transport.send`, then translates `UrpResponse` → `AdapterResponse`. It does not implement `generateStream`; streaming over URP is deferred to the roadmap.

## Agent Call Protocol (ACP)

ACP is a minimal HTTP convention for exposing a server-side agent as a callable endpoint. Unlike URP — which carries full conversation history and tool definitions so a stateless backend can reason — ACP is point-to-point: the caller passes a single input string, and the server returns a single output string. The server-side agent encapsulates its own state, history, and reasoning loop.

### Request

`POST /` (or any path agreed by the developer)

```json
{
  "input": "Summarise the following text: ..."
}
```

| Field   | Type     | Required | Notes                              |
|---------|----------|----------|------------------------------------|
| `input` | `string` | Yes      | The prompt or task for the agent   |

### Response

```json
{
  "output": "The document discusses three key themes..."
}
```

| Field    | Type     | Required | Notes                              |
|----------|----------|----------|------------------------------------|
| `output` | `string` | Yes      | The agent's final response         |

### Error Response

On failure, the server returns a non-2xx status code. An optional JSON body may include a `message` field:

```json
{ "message": "Agent execution failed: model unavailable" }
```

`AcpAgent` catches non-2xx responses and throws `AdapterError` with the status code and message.

### Design Notes

- ACP is intentionally minimal. It carries no conversation history, no tool definitions, and no configuration — those are internal concerns of the server-side agent.
- A backend built with `rust-agent-kit` can expose any `AgentRunner` as an ACP endpoint with a thin HTTP handler (~10 lines).
- Third-party agent APIs that do not speak ACP natively can be wrapped in a custom `AgentExecutor` implementation without modifying `AgentTool`.

## Transport Layer (`src/transport/`)

### `HttpTransport` (`src/transport/http.ts`)

```typescript
export interface HttpTransportOptions {
  url: string;
  headers?: Record<string, string>;  // e.g. { Authorization: 'Bearer ...' }
  signal?: AbortSignal;
}

export class HttpTransport implements UrpTransport {
  constructor(options: HttpTransportOptions);
  send(request: UrpRequest): Promise<UrpResponse>;
}
```

Uses the Fetch API (`fetch`). Throws `AdapterError` on non-2xx status codes or network failures. Compatible with browser, Deno, Cloudflare Workers, and any runtime that exposes the standard `fetch` global.

### `WebSocketTransport` (`src/transport/websocket.ts`)

```typescript
export interface WebSocketTransportOptions {
  url: string;
  protocols?: string[];
}

export class WebSocketTransport implements UrpTransport {
  constructor(options: WebSocketTransportOptions);
  send(request: UrpRequest): Promise<UrpResponse>;
  close(): void;
}
```

Maintains a single persistent WebSocket connection. `send` writes the serialised `UrpRequest` as a text frame and resolves when the matching response frame arrives. Frames are correlated by an auto-generated `request_id` field appended to each request. Falls back to reconnect once on connection loss.

## `PromptApiAdapter` (`src/adapter/prompt-api.ts`)

`PromptApiAdapter` enables fully offline, client-side inference using the [Chrome Prompt API](https://developer.chrome.com/docs/ai/prompt-api), which runs Gemini Nano on-device via the `window.LanguageModel` global (available in Chrome 138+ behind the `#prompt-api-for-gemini-nano` flag, and in Chrome Extensions).

```typescript
export interface PromptApiAdapterOptions {
  /** Sampling temperature. Chrome Extensions only; ignored in web pages. */
  temperature?: number;
  /** Top-K sampling. Chrome Extensions only; ignored in web pages. */
  topK?: number;
  /** Declare expected input modalities to gate availability checks. */
  expectedInputs?: Array<{ type: 'text'; languages?: string[] }>;
  /** Declare expected output modalities to gate availability checks. */
  expectedOutputs?: Array<{ type: 'text'; languages?: string[] }>;
}

export class PromptApiAdapter implements LlmAdapter {
  /** Private — use the static factory. */
  private constructor(options: PromptApiAdapterOptions) {}

  /**
   * Checks availability via `LanguageModel.availability()` before constructing.
   * Throws `AdapterError` if the result is `"unavailable"`.
   * When the result is `"after-download"` or `"downloading"`, passes a
   * `monitor` to `LanguageModel.create()` that emits download progress via
   * the optional `onDownloadProgress` callback.
   */
  static async create(
    options?: PromptApiAdapterOptions & {
      onDownloadProgress?: (loaded: number) => void;
    },
  ): Promise<PromptApiAdapter>;

  generate(request: AdapterRequest): Promise<AdapterResponse>;
  generateStream(request: AdapterRequest): AsyncIterable<AdapterStreamChunk>;
}
```

### Session Lifecycle

The Prompt API maintains conversation state inside a `LanguageModel` session object. `PromptApiAdapter` creates a **new session per `generate()` call** to remain stateless and consistent with the `LlmAdapter` contract (the runner already passes full history on every call):

1. Translate `request.messages` (all but the final user turn) into the Prompt API's `MessageArray` format and pass them as `initialPrompts` on `LanguageModel.create()`.
2. The system prompt (`request.system`) is prepended as a `{ role: 'system', content: '...' }` entry in `initialPrompts`.
3. Call `session.prompt(finalUserMessage, { responseConstraint })` (or `session.promptStreaming(...)` for streaming).
4. Call `session.destroy()` in a `finally` block to free GPU memory immediately after each turn.

### Message Translation

`Message` → Prompt API `MessageArray`:

| `MessageContent.type` | Prompt API role | Prompt API content |
|---|---|---|
| `'text'` | `'user'` / `'assistant'` | plain string |
| `'tool_calls'` | `'assistant'` | JSON-serialised calls (see below) |
| `'tool_result'` | `'user'` | JSON-serialised result (see below) |

Tool call turns are serialised as JSON strings in the message content so the model can see the call/result cycle as part of its context window, even though the Prompt API has no native function-calling protocol.

### Tool Calling Strategy

The Prompt API does not support native function calling. `PromptApiAdapter` uses a **structured-output injection** approach to simulate it:

1. When `request.tools` is non-empty, the adapter appends a tool-use instruction block to the system prompt that lists all available tools with their JSON Schema `parameters`.
2. It passes a `responseConstraint` (a JSON Schema discriminated union) to `session.prompt()` forcing the model into one of two response shapes:

```jsonc
// Final text response
{ "type": "text", "content": "The capital of France is Paris." }

// Tool invocation(s)
{
  "type": "tool_calls",
  "calls": [
    { "id": "call_1a2b3c", "name": "search_web", "args": { "query": "capital of France" } }
  ]
}
```

3. The `responseConstraint` JSON Schema:

```json
{
  "oneOf": [
    {
      "type": "object",
      "properties": {
        "type":    { "const": "text" },
        "content": { "type": "string" }
      },
      "required": ["type", "content"],
      "additionalProperties": false
    },
    {
      "type": "object",
      "properties": {
        "type": { "const": "tool_calls" },
        "calls": {
          "type": "array",
          "items": {
            "type": "object",
            "properties": {
              "id":   { "type": "string" },
              "name": { "type": "string" },
              "args": { "type": "object" }
            },
            "required": ["id", "name", "args"],
            "additionalProperties": false
          }
        }
      },
      "required": ["type", "calls"],
      "additionalProperties": false
    }
  ]
}
```

4. Call `id` values are generated by the adapter (e.g. `call_<nanoid(6)>`) since the Prompt API does not assign them.
5. The adapter parses the response and maps it to `AdapterResponse.text` or `AdapterResponse.toolCalls`.

**Structured output without tools:** When `request.tools` is empty and `request.outputSchema` is set, the `outputSchema` is passed directly as the `responseConstraint`, giving the same structured-output guarantee as the Prompt API's native `session.prompt({ responseConstraint })`.

**Streaming with tools:** `generateStream` falls back to `generate` when `request.tools` is non-empty, because the full `responseConstraint` JSON must be parsed to determine whether the turn is text or tool calls. Streaming is only active on tool-free turns.

### Context Window Overflow

The Prompt API exposes `session.contextUsage` (tokens consumed) and `session.contextWindow` (maximum capacity). If `LanguageModel.create()` or `session.prompt()` throws a `QuotaExceededError`, the adapter catches it and re-throws as:

```
AgentError('Prompt API context window exceeded (requested N tokens, window is M). Trim conversation history before retrying.')
```

### Availability Check

`PromptApiAdapter.create()` calls `LanguageModel.availability(options)` before constructing the session:

| Availability result | Behaviour |
|---|---|
| `"readily"` | Construct immediately |
| `"after-download"` / `"downloading"` | Construct and pass `monitor` to report download progress via `onDownloadProgress` |
| `"unavailable"` | Throw `AdapterError('Chrome Prompt API is not available in this browser.')` |

### Usage Pattern

```typescript
const adapter = await PromptApiAdapter.create({
  onDownloadProgress: (loaded) => console.log(`Model downloading: ${loaded} bytes`),
});

const runner = new AgentRunner(adapter, registry);
const result = await runner.run(agent, 'Summarise this page.');
```

## Web Worker Support (`src/worker/`)

`AgentRunner` executes synchronously on the main thread by default. For long-running or tool-heavy runs, an optional `WorkerAgentRunner` wrapper offloads the loop to a Dedicated Worker.

```typescript
// Main thread
export class WorkerAgentRunner {
  constructor(workerUrl: string | URL);

  run(agent: AgentConfig, input: string, history?: Message[]): Promise<AgentResult>;
  runStream(agent: AgentConfig, input: string, history?: Message[]): AsyncIterable<AgentEvent>;
  terminate(): void;
}
```

The worker script (`src/worker/agent-worker.ts`) hosts an `AgentRunner` configured by the main thread at startup. Tool calls are routed back to the main thread via `postMessage` and awaited before the worker continues the loop. This keeps tool execution in the main thread context (where DOM access is valid) while the loop overhead runs off-thread.

**Message protocol (internal, not URP):**

| Direction          | Message type         | Payload                          |
|--------------------|----------------------|----------------------------------|
| Main → Worker      | `init`               | `{ adapterConfig, agentConfig }` |
| Main → Worker      | `run`                | `{ runId, input, history }`      |
| Worker → Main      | `tool_call_request`  | `{ runId, callId, name, args }`  |
| Main → Worker      | `tool_call_result`   | `{ runId, callId, result }`      |
| Worker → Main      | `agent_event`        | `{ runId, event: AgentEvent }`   |
| Worker → Main      | `run_done`           | `{ runId, output }`              |
| Worker → Main      | `run_error`          | `{ runId, error }`               |

## Bundle Size Budget

The core bundle (`index.ts` tree, excluding `worker/agent-worker.ts`) must remain under **20 KB gzipped**. The worker entry point is loaded separately and does not count toward the page-load budget.

## Testing Strategy

- **Unit tests** in `tests/unit/` use a stub `LlmAdapter` that returns scripted `AdapterResponse` sequences. They verify loop logic, tool dispatching, event ordering, concurrency, and error propagation without any network calls.
- **Integration tests** in `tests/integration/` start a local HTTP server that speaks URP and exercise the full `AgentRunner → UrpAdapter → HttpTransport → server` path.
- All public types and functions must have TSDoc comments. Examples in doc comments are valid TypeScript and verified by `tsd` or `ts-jest` in `noEmit` mode.

## Roadmap

The following capabilities are planned but not yet implemented:

1. **Streaming over URP.** `UrpAdapter` does not implement `generateStream`; the backend emits the full response in one shot. A streaming variant using server-sent events (SSE) or WebSocket framing would enable incremental `text_delta` events in hybrid mode.
2. **`PromptApiAdapter` streaming with tools.** Currently falls back to non-streaming when tools are present, because the full JSON response must be parsed to distinguish text from tool calls. A future approach could use `session.promptStreaming()` and buffer until a structural boundary is detected.
3. **Stateful `PromptApiAdapter` session option.** The current design creates and destroys a session per `generate()` call. An opt-in stateful mode that reuses a single session across turns (using `session.append()` to add tool results) would reduce per-turn latency on-device by avoiding context re-processing.
4. **Structured output enforcement.** `AgentConfig.outputSchema` is forwarded to adapters but only `UrpAdapter` and `PromptApiAdapter` enforce it. A post-processing validation step (using `ajv`) would give consistent behaviour across all adapters and catch cases where the model ignores the schema.
5. **WorkerAgentRunner configuration API.** The current design initialises the worker with a single adapter config. A richer API would allow hot-swapping the adapter or registry without restarting the worker.
6. **Resumable runs.** Serialising a mid-loop `RunBuilder` state (history snapshot + pending tool call IDs) to JSON would allow a run interrupted by a page reload to be resumed in a new session.
